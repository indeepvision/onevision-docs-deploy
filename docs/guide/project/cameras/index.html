Cameras

Define the cameras that will be used in the project in the section Cameras. Configure here all the features of the cameras and check their status in real time.

Add camera popup​

To add a camera, click on the button in the top left corner.

Controller name​

The name of the camera. Must be unique and is used across the application to identify the camera. It can be changed later.

Controller type​

There are two built-in controller types: GenTL and Emulation.

  • GenTL is a standard controller for GigE Vision cameras.

  • Emulation is a controller that can be used to test the application without a real camera. It generates a synthetic image that can be configured in the Emulation section.

Additional controller types can be added by installing plugins.

Producer​

Once a controller type is selected, multiple producer options might be available, which correspond to different camera manufacturers or different data transmission protocols (transport layer).

  • For the GenTL controller the producers are normally installed with the camera drivers. The producers are DLL files with extension .cti that implement the functionality of the camera for a specific vendor. OneVision will obtain the list of producers from the environment variable GENICAM_GENTL64_PATH (do not confuse it with GENICAM_GENTL32_PATH, for 32-bit applications). This environment variable is normally populated by the camera drivers installer.

  • For the Emulation controller there is only one producer available, which is Emulation.

Vendor​

Read-only field that shows the vendor (or manufacturer) of the producer (not the camera, though in many cases they are the same).

Transport layer​

Read-only field that shows the transport layer of the camera, which is the transmissiong protocol, corresponding to the selected producer. Available Transport layer values are:

Transport layerDescription
GigeGigE Vision (GEV)
Usb3USB3 Vision (U3V)
Usb2USB Video Class (UVC)
EmulationInternal emulation type
UnknownUnrecognized type or various transport layers for the same producer

Status panel​

Once the camera is added, it will appear as a new tab in the top bar. You can delete a camera any time by clicking on the delete button in the top right corner of the camera tab.

You can check the state of the camera controller in the status panel in the top of the screen. There are 4 possible states:

  • Not initialized: The controller has been added but changes have not been saved yet.
  • Disconnected: The controller is not connected to the device.
  • Waiting for connection: The controller has not been able to connect to the camera, and now is retrying to connect.
  • Connected: The controller is connected to the physical camera and ready to be used.

Control panel​

Once the controller is connected to the camera, you can use the Control panel.

  • Connect/disconnect: Switch the connection with the camera on/off.
  • Start/stop acquisition: Start or stop the streaming from the camera, with the configured parameters.
  • Software trigger: Send a software trigger to the camera. This is only available if the camera supports software triggering.
  • Single shot: Capture a single image from the camera.
  • Start/stop video: Start or stop the continuous streaming of images from the camera (video mode). This might not be supported by all cameras.

Device selection​

The device selection sub-menu is used to select the camera device we want to connect the controller to.

Once the camera is connected, the read-only fields in the sub-menu will be populated with information about the camera.

Controller type​

Read-only. The type of controller as indicated in the Add camera popup.

Producer​

Read-only. The producer of the camera as indicated in the Add camera popup.

Vendor​

Read-only. The vendor/manufacturer of the camera is read from the camera itself once it is connected.

Transport layer​

Read-only. The transport layer of the camera is read from the camera itself once it is connected.

Camera serial​

To select the device, enter the in this field with serial number of the camera. The serial number of the camera is normally located on the camera itself.

For Emulation cameras, the serial number is one of the following possibilities: EMU1, EMU2, EMU3, EMU4, EMU5, EMU6, EMU7, EMU8.

Device model​

Read-only. The model is the product identification of the camera established by the manufacturer. It is read from the camera itself once it is connected.

Scan type​

Read-only. The scan type can be one of the following:

  • Areascan: The camera captures the whole image at once.
  • Linescan: The camera captures the image line by line.

This features is read from the camera itself once it is connected.

Basic configuration​

The basic configuration sub-menu is used to configure the parameters of the camera controller itself, not the internal features of the camera as a device. In summary, here you can configure how OneVision and the camera will work together.

Friendly name​

The friendly name is the unique name of the camera controller that will be used throughout the application. It must be unique, but can be changed at any time.

Enable/disable camera​

This option is used to enable or disable the camera controller. If the controller is disabled, it will not be used in the project and will not connect to the camera device.

Critical​

This option is used to mark the camera controller as critical. If the controller is critical, the application will consider an error if the camera is not connected. For example, runtime execution will stop if connection with the camera device is lost.

When a critical camera is not connected, the camera tab will display a "red" icon next to the camera name. If it is not connected but it is not critical, the icon will be "orange" to indicate a warning.

Pixel format​

The pixel format configured here will determine the format of the image that is received from the camera. This might not be the pixel format that camera device delivers to the computer, since OneVision will convert the image to the configured format if necessary.

It might be necessary for example to convert an image that is streamed from the camera in Bayer format to BGR format, so that it can be used by the AI models.

Only 3 possible formats are allowed to stream from the camera:

  • Mono8: Monochrome 8-bit
  • Mono16: Monochrome 16-bit
  • BGR8: Blue-Green-Red 8-bit

Consult the supported image formats by images in the Specifications section.

tip

Like OpenCV, OneVision uses the BGR8 format instead of the RGB8 format. This means that the first pixel in the image is the blue component, the second pixel is the green component and the third pixel is the red component.

Rotation​

The rotation of the image can be configured here. The rotation is applied to the image after it is received from the camera, by software. The user will receive the image already rotated.

Rotation can be configured with the following values:

  • 0: The image is not rotated (default).
  • 90: The image is rotated 90 degrees clockwise.
  • 180: The image is rotated 180 degrees clockwise.
  • 270: The image is rotated 270 degrees clockwise.
tip

If the camera's internal features allow rotation and flip, it might be more efficient to configure the rotation directly in the camera device instead of configuring it by software here in OneVision.

Flip​

The flip of the image can be configured here. The flip is applied to the image after it is received from the camera, by software. The user will receive the image already flipped.

Flip can be configured with the following values:

  • None: The image is not flipped (default).
  • Horizontal: The image is flipped horizontally.
  • Vertical: The image is flipped vertically.
  • Both: The image is flipped horizontally and vertically.

Image buffer size​

The image buffer is a ring buffer, which means that it has a fixed size and it is filled with the images that are received from the camera. Once the buffer is full, the oldest images are discarded to make room for the new images.

For many applications the image buffer will have no importance, it can be left with the default value.

However, for some applications it might be important to configure the size of the buffer. For example, if the camera and processing work in parallel, not at the same speed, it might be necessary to increase the size of the buffer to avoid losing images if processing is slower than acquisition.

Log level​

This option is used to configure the level of detail of the logs that are generated by the camera controller. This option is mainly intended for developers. The possible values are:

  • DEBUG: The most detailed level of logs.
  • INFO: A normal level of logs (default).
  • WARNING: Only warnings and errors are logged.
  • ERROR: Only errors are logged.

Features list​

In this part of the configuration you can configure all the internal features that the camera supports.

The features are normally grouped by categories, which are shown in the left-side panel. The categories are defined by the camera manufacturer, and they are not necessarily the same for all cameras, although GeniCam compliant cameras follow the EMVA's Standard Feature Naming Convention (SFNC).

The camera features shown here are written to the camera with the values desired by the user, and then are read back from the camera to check that the values have been written correctly. It might happen that the exact value introduced by the user is not allowed.

The features list is read from the camera once it is connected, then the features are stored in the project configuration. You need to connect at least once to the camera to be able to retrieve the features.

Image buffer​

Here we will describe how the camera images are stored in the image buffer. First of all, the size of the buffer is configured in the Image buffer size parameter.

The image buffer is actually composed of 2 buffers, the input buffer and the output buffer. The sum of both buffers is the total size of the image buffer.

  • Input buffer: These are the image containers that will be written by the camera when new images are received. They are the "available images".
  • Output buffer: These are the image containers that have already been filled, and are waiting to be retrieved by the user. They are "locked" until they are retrieved.

In the "status panel" you can monitor in real time the number of image containers available in each buffer. The total number of image that have arrived is shown in the Image count field.

This is how the buffers work:

  1. When the camera is first connected, empty image containers are created and placed in the input buffer.
  2. When a new image arrives from the camera, an empty container is taken from the input buffer, is written with the data, and then is placed in the back output buffer.
  3. When the user requests an image, the frist image (the oldest) in the output buffer is taken and returned to the user. The image container is then removed from the output buffer and placed in the input buffer.
  4. If the camera is streaming images faster than the user is retrieving them, the input buffer will be emptied until there is one last image left. At this point, images are automatically discarded from the output buffer and returned to the input buffer to make room for the new images.
  5. In video mode, only the last image in the output buffer is kept, the rest are discarded, so the output buffer size will always be 1.
